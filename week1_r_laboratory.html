Laboratory Overview

Welcome to your first Weekly R-Laboratory. This course operates on a Code-Along Philosophy: you are not expected to build complex programming architecture from scratch. Instead, you will interact with standardized structural scaffolding, learning how to manipulate key parameters, clean data pipelines, and interpret empirical outputs to drive strategic, evidence-based health administration and leadership.

Today’s session is divided into three progressive parts:

  1. Part I: The Excel Friction Exercise: Exposing the limitations of spreadsheets.

  2. Part II: The R Building Blocks: Mastering the 4 basic rules of coding syntax.

  3. Part III: The Preprocessing Pipeline: Working with a real-world clinical dataset.

  4. Part IV: The Value-Based Storyline: Calculating, visualizing and reporting value.

Part I: The Excel Friction Exercise

Background & Objective

Before diving into code, we must evaluate the “value proposition” of transitioning from an Excel environment to an R workspace. In healthcare administration, manual data tasks introduce non-reproducible errors, hidden risks, and productivity costs.

The Task

Navigate to your Canvas module and download the 20-row spreadsheet sample named blood_storage_sample.csv. Open it in Microsoft Excel and attempt to execute the following administrative actions:

  1. Variable Organization: Locate the column named TimeToRecurrence. Manually change the column header name to months_to_return.

  2. Indicator Engineering: Create a new column named Age_PSA_Ratio. Write a cell formula to calculate this metric by dividing the patient’s Age by their PreopPSA value for all rows.

  3. Subsetting: Use Excel’s filtering feature to isolate only high-risk patients who are Age 65 or older AND have a baseline PSA count of greater than or equal to 10.

Executive Reflection

Once complete, consider these three operational challenges:

  1. The Scale Problem: If you receive an updated system database next week containing 200,000 patient rows instead of 20, how long will it take to manually re-click, re-formula, and re-filter the spreadsheet?

  2. The Black Box Problem: If you accidentally modify a cell value while scrolling, does Excel provide an immutable alert trail, or does the model break quietly?

  3. The Audit Problem: If a regulatory body requests the exact logical sequence you used to clean this data, can they inspect or reproduce your mouse clicks?

Part II: The R Building Blocks

Building Bock 1: The Assignment Operator (<-)

In Excel, data is permanently bound to a specific grid cell (like B4). In R, we store data inside a named, flexible object using the assignment arrow (<-). Think of this as placing data inside a labeled container on an executive asset shelf.

Building Bock 2: Data Types (Numeric vs. Character Strings)

R enforces strict structural rules based on data classification to preserve math integrity.

  1. Numeric: Raw numbers that R can perform calculations on (e.g., fractional laboratory counts, financial budgets).

  2. Character Strings: Text data used for descriptive labels. Text values must be wrapped in quotation marks ("") so R recognizes them as text rather than a programmatic command.

Code 
# Numeric indicator
baseline_hb <- 14.2

# Character string label
facility_name <- "Schulich Health Network"

Building Bock 3: Logical Operators (>=, &, ==)

This is the foundational logic used to filter populations. R evaluates clinical rows against your conditions and returns a binary TRUE or FALSE assessment.

  1. >= : Greater than or equal to
  2. & : And (both conditions must resolve to true simultaneously)
  3. == : Is exactly equal to (we use a double equals sign for evaluation, because a single equals sign is reserved for setting parameters inside functions)
Code 
# Evaluate if our stored patient age meets a clinical threshold
patient_age >= 70
[1] FALSE

Building Bock 4. Missing Values (NA)

In Excel, missing observations are represented by blank cells, which can easily be calculated as zeros by mistake, heavily skewing your data summaries. R explicitly flags missing records as NA (Not Available), forcing decision-makers to intentionally choose how to address incomplete metrics.

Part III: The Preprocessing Pipeline

Data preprocessing consumes up to 80% of any healthcare analytics project. We will now utilize the medicaldata library to build an automated, fully reproducible cleaning pipeline that transforms a raw, messy dataset into an executive matrix.

Code-Along Instructions

Copy the code pipeline below into your script file:

Code 
# ACTIVATE PROGRAMMATIC SUITES (Load Libraries)
library(tidyverse)    # Collection of packages for data cleaning, wrangling, and plotting
library(medicaldata)  # Provides verified, native real-world medical teaching datasets
library(conflicted)   # Handles name clashing between libraries
library(gt)           # Generates publication-quality executive summary tables

# DECLARE PACKAGE PREFERENCES (Resolve Hidden Pkg. Conflicts)
# We explicitly tell R to use 'dplyr' for commands that exist in multiple packages
conflict_prefer("filter", "dplyr") # Forces R to use the data-filtering tool from dplyr
conflict_prefer("lag",    "dplyr") # Forces R to use the time-lagging tool from dplyr

# INTRODUCE THE CLINICAL SOURCE FILE (Load Data)
raw_data <- medicaldata::blood_storage # Extracts the prostate oncology dataset

# BUILD THE REPRODUCIBLE DATA ASSEMBLY LINE
cleaned_executive_data <- raw_data %>%
  
  # Pillar A: Structure & Selection
  
  # Isolate only the specific clinical metrics required for our strategic tracking
  select(Age, PreopPSA, TimeToRecurrence, Recurrence) %>%
  
  # Standardize native database headers into clear corporate snake_case descriptors
  rename(
    patient_age      = Age,
    baseline_psa     = PreopPSA,
    months_to_return = TimeToRecurrence,
    return_status    = Recurrence
  ) %>%

  # Data Type Cleaning & Categorization
  # Convert a numeric code into text so R treats it as a grouping label, not a math value
  mutate(
    return_status = as.character(return_status)
  ) %>%

  # Pillar B: Data Transformation & Value Engineering
  
  # Formulating the Lecture 1 Value Equation: Outcomes (Months Disease-Free) / Baseline Burden (PSA)
  mutate(
    systemic_risk_ratio = patient_age / baseline_psa,
    healthcare_value_index = months_to_return / baseline_psa
  ) %>%
  
  # Pillar C: Subsetting & Filtering
  # Isolate only patients who are age 65 or older AND have a baseline PSA score of 10+
  filter(patient_age >= 65 & baseline_psa >= 10)

stakeholder_summary <- cleaned_executive_data %>%
